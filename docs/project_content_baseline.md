# Project Content Baseline — 工程作品集项目总结与证据凝练

> 本文档约束项目资料审计、事实提取、工程叙事、证据表达、素材编排和内容验收。
> `UI_baseline.md` 负责视觉呈现，`engine_baseline.md` 负责类型、路由、组件、构建与部署；项目事实与文案以本文档为准。

## 1. 文档目标

作品集中的每个项目都应是一份可快速浏览、也经得起追问的工程案例，而不是 README 摘要、简历条目扩写或技术栈广告。

内容最终需要回答六个问题：

1. 项目为什么产生，原始约束是什么？
2. 当时真正难以解决的矛盾是什么？
3. 项目经历了怎样的演进，关键决策为何发生？
4. 本人在其中具体设计、实现、验证和交付了什么？
5. 哪些代码、提交、测试、日志、部署、Release 或实机记录可以支撑这些表述？
6. 当前结论适用于什么范围，是否存在必须对外说明的误读风险？

合格的项目叙事应形成以下链路：

```text
起因与约束 → 核心矛盾 → 方案演进 → 个人决策与实现
            → 结果与证据 → 必要的范围校准（按需） → 当前路线
```

首页负责让读者判断是否值得继续阅读；项目详情负责解释工程过程、贡献与证据。两者使用同一份 `Project` 数据，详略可以不同，事实口径不能不同。

## 2. 内容工作的基本原则

### 2.1 先审计，后写作

不得只根据交接 Prompt、旧简历或 README 直接生成最终文案。它们是调查入口，不是天然正确的最终事实。

写作前至少需要核对：

- 当前分支、HEAD、提交历史与作者归属；
- README、设计文档、阶段总结、测试报告和原始日志；
- 关键源码、配置、构建脚本与部署拓扑；
- 上游仓库、Fork 链路、许可证与第三方组件；
- 当前测试、构建、线上部署或实机运行状态；
- 图片、视频和日志对应的版本、日期与环境；
- 用户补充的项目初衷是否能被历史文档、提交或代码演进印证。

如果资料之间发生冲突，不拼接成一个更强结论，也不擅自替创作者解释。应定位冲突对应的版本和环境，缩小表述范围，并在必要处注明日期、commit 或验证条件。

### 2.2 写工程判断，不写功能清单

项目价值优先来自：

- 识别了什么结构性问题；
- 如何拆分职责、状态和边界；
- 为什么采用当前方案；
- 方案解决了什么，又引入了什么代价；
- 如何用测试、日志、部署或实机操作验证。

“实现登录、聊天、文件树、轨迹控制”等功能只能说明做过什么，不能替代架构判断和个人贡献。

### 2.3 事实强度不得超过证据强度

以下证据不能互相替代：

```text
源码存在 ≠ 构建通过
构建通过 ≠ 功能通过
单元测试通过 ≠ 端到端通过
mock / fake / dry-run 通过 ≠ 真实服务或硬件通过
部署可访问 ≠ 核心链路全部验证
单次实机记录 ≠ 长期稳定性验收
固定场景压测 ≠ 系统容量上限或通用 SLA
Release 存在 ≠ 生产可用
路线规划 ≠ 已实现能力
```

### 2.4 个人贡献必须可定位

“核心开发”“主要开发”不能单独作为证据。需要进一步说明：

- 负责的模块或链路；
- 对应的提交、代码目录、设计文档或发布记录；
- 本人做出的关键决策；
- 与团队成员、上游项目和第三方组件的边界。

提交数量可以辅助说明持续投入和作者归属，但不能代替功能正确性、工程质量或外部影响。

### 2.5 边界只用于阻止必要的误读

作品集默认展示项目做成了什么，而不是主动列出没有实现什么。`boundaries` 是可选字段，不是每个项目都必须出现的“免责声明”。

只有当当前文案容易让读者形成一项明显错误、且会影响工程能力判断的结论时，才补充范围校准，例如：

- 压测来自固定拓扑，不能直接外推为系统容量上限；
- 结果来自 mock / dry-run，不能解释为完整实机验收；
- 项目基于上游演进，需要区分继承能力与个人贡献；
- 研究结果不支持医疗、临床或因果结论；
- roadmap 中的能力尚未进入当前实现。

如果标题、摘要和贡献描述本身已经准确，就直接省略 `boundaries`。不得为了显得严谨，主动罗列缺少的高级功能、普通技术债、无关 UI 问题、构建环境差异或内部待办；这些内容会把读者注意力从项目价值引向不必要的追问。

## 3. 证据来源与事实台账

### 3.1 证据来源优先级

每条候选结论都应寻找与其直接对应的证据。通常按以下顺序判断：

1. **直接运行证据**：本轮构建、测试、浏览器操作、接口响应、数据库记录、设备输出或实机操作；
2. **版本化工程证据**：具体 commit、源码路径、测试报告、压测原始日志、Release、部署记录；
3. **外部可核验记录**：合并 PR、公开仓库、公开 Release、线上可访问服务、上游引用；
4. **项目文档**：设计说明、阶段总结、README、实验记录；
5. **创作者陈述**：用于还原初衷和隐性背景，并继续用代码、历史或记录交叉验证；
6. **计划文档**：只能支撑 roadmap、research question 或 future work。

“当前代码”和“历史实机记录”可能都是真实的，但回答的是不同问题。不得因为当前 HEAD 已重构，就把旧拓扑的实机记录自动视为新架构的实机验证。

### 3.2 事实台账

正式写入 `projects.ts` 前，建议先建立项目事实台账：

| 候选结论 | 版本 / 日期 | 环境与条件 | 证据位置 | 可使用措辞 | 必须附带的边界 |
|---|---|---|---|---|---|
| 示例：测试通过 | 当前 HEAD / 日期 | OS、依赖、是否 mock | 命令输出或报告 | `N/N 测试通过` | 测试类型与未覆盖范围 |
| 示例：实机闭环 | 固件 / 配置版本 | 硬件拓扑与操作路径 | 日志、照片、视频 | `在该拓扑下完成实机验证` | 不外推到其他拓扑或稳定性 |
| 示例：性能指标 | commit / 日期 | 并发、间隔、网络、接口 | 报告与原始日志 | 精确数值和分位数 | 不代表容量上限 |

无法填写“版本、条件、证据位置”的数字，默认不进入首页指标；必要时只能作为范围较弱的项目说明。

### 3.3 验证状态必须分层

内容中常用的验证层级如下：

| 层级 | 可以证明 | 不能自动证明 |
|---|---|---|
| 代码 / commit 可定位 | 确实实现或修改过某条路径 | 可以构建、运行或稳定工作 |
| 静态检查 / 构建 | 当前环境下能够通过编译或打包 | 业务行为正确 |
| 单元测试 | 被断言的局部逻辑正确 | 服务集成、真实设备和用户流程 |
| mock / fake / dry-run | 控制流程、错误分支或合同逻辑 | 真实网络、传感器、电机和物理时序 |
| 端到端操作 | 给定环境下完整业务路径可走通 | 大规模、长期或多拓扑稳定性 |
| 真实硬件 | 指定硬件、配置和动作得到验证 | 所有硬件组合与长期可靠性 |
| 外部合并 / 发布 / 采用 | 存在第三方验收或公开交付 | 任意质量等级或商业生产能力 |

## 4. 项目审计工作流

### 4.1 建立项目上下文

开始前记录：

- 本地项目目录与公开仓库 URL；
- 目标分支、当前 HEAD 和主要历史分支；
- 关键总结文档、实验报告、运行说明和素材目录；
- 是否为团队项目、Fork、课程项目、比赛项目或研究项目；
- 可运行的服务、测试环境、硬件环境和已知安全限制；
- 用户最希望突出、同时最担心被夸大的内容。

只读审计不授权修改源项目。若运行会改动真实数据库、设备状态或外部服务，应优先使用隔离环境、测试账号和项目已有脚本；无法安全验证时，保留已有记录并降低结论强度。

### 4.2 阅读顺序

推荐顺序：

1. 查看仓库状态、分支、HEAD、作者和时间线；
2. 阅读根 README 与关键总结文档，建立初始模型；
3. 通过源码和配置核对核心链路，不只搜索关键词；
4. 阅读测试、压测、实验、部署与实机记录；
5. 核对上游历史和本人修改边界；
6. 运行当前可安全复现的构建与测试；
7. 条件允许时操作真实前端、API、服务或硬件；
8. 整理事实台账，再开始写文案和挑选图片。

### 4.3 运行审计记录

每次新增运行结论，应记录：

- 日期；
- commit / 版本；
- 操作系统、核心依赖和部署拓扑；
- 实际运行命令或入口；
- 通过项、受限项和失败项；
- 浏览器控制台、服务日志、数据库或设备状态；
- 截图和日志的归档位置。

审计项使用两种状态：

- `passed`：本轮按所述环境实际完成，结果与声明一致；
- `qualified`：只完成部分检查、沿用历史证据或存在会改变解释范围的限制。

不得为了让项目页面更完整，把没有执行的检查标为 `passed`。

### 4.4 收尾

运行结束后应关闭临时进程，避免遗留服务、测试账号、敏感令牌或设备状态。公开素材不得包含密码、Token、私有 IP、真实患者信息、个人隐私或无关桌面内容。

## 5. 统一叙事结构

### 5.1 项目身份

项目头部需要用最少信息建立准确预期：

- `title`：仓库或产品的稳定名称；
- `subtitle`：项目对象 + 核心技术或核心能力，不罗列全部 Stack；
- `category`：按问题域选择 `systems / ai / embedded / robotics`，允许多分类；
- `role`：本人角色和贡献性质，团队项目必须显式说明；
- `period`：真实起止时间；
- `status`：发布、已验证、MVP、持续迭代或进行中，不由仓库版本号自动推断；
- `featured`：只影响首页呈现，不代表证据等级或项目高低。

`systems` 在界面中表达 System Engineering / 系统与后端方向，不等同于只收录 C++；`ai` 用于模型、检索、知识图谱和研究工程，也可与 `systems` 同时存在。

### 5.2 一句话摘要 `summary`

摘要建议用一到两句话完成：

```text
在什么约束下，为解决什么核心矛盾，采用了哪些关键机制，形成了什么可验证结果。
```

摘要不应：

- 从“本项目基于 X、Y、Z 技术”开始；
- 写死作品集当前项目数量或筛选结构；
- 使用“功能完善、性能优异、稳定可靠”等无证据判断；
- 把未来路线塞进当前成果；
- 复制完整背景、贡献列表和指标。

### 5.3 背景与问题 `background`

背景应描述项目真正的起点，而不是抽象行业介绍。优先回答：

- 原系统、上游方案或最初实现是什么状态？
- 哪个具体限制阻止了目标继续推进？
- 为什么简单增加条件分支、功能或配置不能解决？
- 这项限制对用户流程、系统语义或工程维护造成什么影响？

如果项目经历多次重构，背景必须保留时间顺序。不能把最终架构结论倒写成项目最初就已明确的问题。

### 5.4 方案与贡献 `solution` / `highlights`

`solution` 解释整体方案和演进，`highlights` 提炼 4–6 条个人贡献。每条贡献尽量包含：

```text
动作 + 作用对象 + 关键机制 + 工程结果
```

优先使用“拆分、收敛、抽象、实现、迁移、治理、验证、归档”等可定位动作，少用“负责、参与、深入研究”等无法核验的角色词。

方案中需要明确区分：

- 第一版快速验证与后续架构重构；
- 接口层、业务层、状态层和硬件执行层；
- 正常链路与失败恢复链路；
- 实现能力与验收能力；
- 个人实现与上游继承。

### 5.5 架构与决策 `architecture`

只有当流程、状态或架构取舍能明显帮助理解项目时才使用，不为每个项目强行补架构图。

`flow` 应描述一条真实端到端链路，每一步包含输入、职责或状态变化，避免把技术名称排成流水账。

每项 `decision` 回答：

- `context`：旧方案或当时约束；
- `decision`：具体做了什么选择；
- `outcome`：解决了什么；
- `tradeoff`：引入了什么代价、风险或未验证部分；
- `evidence`：源码、文档、commit 或报告位置。

通常保留 3–5 个真正影响系统形态的决策。普通组件选择、依赖安装和样式调整不进入架构决策。

### 5.6 结果与指标 `results` / `metrics`

`results` 按“结果—条件—证据—解释范围”组织。`metrics` 选择最多三个首页值得快速扫描的信号。

指标必须包含：

- 值与单位；
- 测试对象；
- 核心条件；
- 错误数、成功率或分位数等必要上下文；
- 日期或 commit（正文或 evidence 中提供）；
- 不可外推的范围。

优先级一般为：真实业务 / 实机结果 > 端到端闭环 > 测试与质量结果 > 数据规模 > 发布与提交数量。项目性质不同可以调整，但不应为了凑满三个指标加入无意义代码行数。

### 5.7 证据摘要 `evidence`

`evidence.items` 是读者进入正文前的事实索引，通常保留 5–8 条，覆盖：

- 仓库、分支、commit 或 Release；
- 当前构建或测试结果；
- 核心端到端、线上或实机记录；
- 最重要的性能或规模数字；
- 关键文档、原始日志或外部合并记录。

项目级 `EvidenceLevel` 是整体可核验程度的摘要，不替代逐条证据：

| 等级 | 使用条件 |
|---|---|
| `A` | 有明确第三方验收、合并、采用或其他外部可验证结果 |
| `B` | 公开仓库、版本化测试、压测、Release、部署或实机记录可核对 |
| `C` | 经历与实现真实，但主要证据仍在本地或公开材料不足 |
| `D` | 主要处于路线、设计或尚未验收阶段 |

等级不由提交数、测试数简单计算；同一项目即使为 `B`，仍要逐项说明哪些是实机、哪些是 dry-run。

### 5.8 协作、研究与上游

以下字段按项目性质选用：

- `collaboration`：团队目标、本人角色、具体负责范围与其他成员边界；
- `research`：研究问题、当前状态、本人职责、现有产出与尚未得到支持的结论；
- `upstream`：Fork 链路、上游已有能力、本人新增能力与许可证归属。

团队项目不能因为大部分 commit 来自本人，就省略团队背景；Fork 项目不能因为重构规模较大，就把上游协议、框架或基础能力写成从零实现。

### 5.9 边界与路线 `boundaries` / `roadmap`

`boundaries` 按需使用，简单项目可以完全省略。准备增加一条边界前，应先回答：

1. 当前页面是否真的会诱导读者得出错误结论？
2. 这个误读是否会影响目标岗位对项目能力的判断？
3. 能否先通过更准确的标题、摘要或贡献归属消除误读？

只有前两项为“是”、且第三项无法解决时，才保留边界。通常控制在 1–4 条，并与最强声明直接对应，例如：

- 性能指标对应测试拓扑与负载边界；
- AI 结果对应数据集、评测方法和非临床边界；
- 控制逻辑对应 mock 与实机差异；
- 新架构能力对应尚未完成的组合验收；
- 上游能力对应个人贡献边界。

不要使用边界章节枚举“没有 LSP、没有插件、没有 CI、没有跨平台”等缺失项，除非页面正文正在明确声称具备这些能力。对普通练习项目，准确说明“基于什么、实现什么、交付什么”通常已经足够。

`roadmap` 只记录真实存在的阶段：

- `done`：有实现和相应证据；
- `current`：正在实现或验收；
- `planned`：后续方向。

路线中的 ROS2、视觉、VLM、策略学习等技术不得提前进入已完成 `highlights` 或当前 Stack；如确有环境预研，应明确写成预研或规划。

### 5.10 技术栈与链接

`stack` 按工程域分组，说明项目真正使用的技术，而不是关键词集合。建议优先使用：

- 语言与构建；
- 网络、通信或协议；
- 存储与数据；
- 控制、固件或硬件；
- 前端与可视化；
- 测试、评测、压测与部署。

`links` 优先指向具体分支、commit、报告、架构文档、Release 或 PR。只有仓库首页时才只链接根仓库。所有链接必须能够支撑其标签含义。

## 6. 不同项目类型的专项要求

| 项目类型 | 必须补充的事实 | 常见错误外推 |
|---|---|---|
| 系统 / 后端 | 拓扑、协议、状态语义、错误路径、压测条件、p95 / p99 与错误数 | 单接口压测写成整套系统容量；单节点写成分布式生产能力 |
| AI / GraphRAG | 数据来源与规模、检索和生成链路、模型 / 服务、评测集、引用与降级行为 | 规则通过写成质量提升；研究问答写成诊断或临床产品 |
| 嵌入式 | MCU / 板卡、构建目标、RAM / Flash、通信协议、调试方式、实机范围 | 编译通过写成整车或设备验收；demo 数量写成全部测试通过 |
| 机器人 | 硬件自由度、总线、控制频率、轨迹与限速、仿真 / fake / 真机层级 | fake robot 单测写成真机安全；规划中的 ROS2 / 视觉写成已完成 |
| Fork / 下游项目 | 上游链路、上游已有功能、个人改动、许可证、兼容策略 | 把协议、框架或原始 UI 写成个人从零开发 |
| 桌面工具 / IDE | 编辑器或 GUI 框架、系统集成点、Release、平台与实际功能范围 | 集成 QScintilla 等组件写成自研内核或大型 IDE |
| 开源贡献 | 上游问题、补丁范围、review / 合并状态、版本与实际影响 | PR 提交写成已采用；局部修复写成主导整个项目 |

专项要求用于决定“还要查什么”，不是给页面套固定话术。

## 7. 素材选择与编排

### 7.1 每张素材必须有职责

素材主要承担四种作用：

1. **建立认知**：实物、产品主界面、最终输出；
2. **解释系统**：架构图、控制链路、数据流、配置结构；
3. **证明结果**：测试报告、Dashboard、终端结果、数据库或设备状态；
4. **还原使用**：连续页面、真实操作路径、部署或实机过程。

无法说明“这张图证明什么”的素材，不应仅为增加数量进入项目页。

### 7.2 字段分工

| 字段 | 素材职责 |
|---|---|
| `cover` | 最能代表项目的单张首图，也是项目列表首图 |
| `showcaseMedia` | 详情首屏轮播，补足核心界面、实物或输出，通常精选 2–4 张 |
| `architecture.media` | 只服务于架构链路或关键模型解释 |
| `evidenceMedia` | 放在结果附近，展示压测、评测、引用、运行结果等证据 |
| `audit.media` | 与本轮操作审计直接相关的界面、状态或过程 |

首屏不承担“展示全部截图”的任务。其余图片应放在对应文字附近，而不是在页面末尾无差别平铺。

### 7.3 图片说明

- `alt` 描述画面实际内容，供可访问性和图片加载失败时使用；
- `caption` 说明版本、环境、操作或它能够支持的结论；
- 历史版本、内测页面、mock 环境和旧拓扑必须在 caption 中注明；
- 截图不能通过裁切隐藏会改变结论的重要失败状态；
- `fit: contain` 用于需要完整阅读的界面和架构图，`cover` 用于允许裁切的氛围或实物画面；
- `position` 只用于保护主体，不通过构图制造与事实不符的重点。

### 7.4 素材工程处理

- 原始截图和照片存入被 Git 忽略的 `temp/`；
- 发布目录只保留选中的 WebP / PNG 等 Web 素材；
- 在不破坏文字清晰度的前提下压缩体积；
- 清理不需要的 EXIF、GPS 和设备信息；
- 文件名稳定、可读，不使用空格和随机序号；
- 检查桌面、平板和 390px 移动视口下的主体裁切；
- 图片点击放大即可，不用把“放大图片”写成额外内容动作。

## 8. 已完善项目形成的质量参照

这些案例用于说明方法，不要求后续项目复制同样的篇幅或字段。

### 8.1 MyChat：运行语义与带条件指标

MyChat 的有效叙事不是“实现了聊天系统”，而是说明 Gateway 为什么需要统一 HTTP / WSS packet 边界、远程 Push 为什么必须回到连接所有者、持久化与在线投递分别承担什么语义。性能数字同时保留并发、发送间隔、p95、错误数和跨主机单路径边界；当前六进程操作审计与历史 CTest、压测报告分开陈述。

后续系统项目应达到相同要求：不仅展示结果，还要解释链路、失败路径和数字适用范围。

### 8.2 ASD-KGRAG：系统、研究与产品边界分层

ASD-KGRAG 将当前系统实现、线上部署实测、50 题评测、团队角色和后续 VLM / 论文路线拆成不同层次。Dashboard 数字、Ask 页面响应和本地 unittest 各自注明来源，不把检索诊断或规则检查直接写成回答质量提升，也不把研究资料问答写成医疗诊断能力。

后续 AI 项目必须说明数据、评测与研究结论之间真正成立的关系。

### 8.3 Colorful-U1：真实演进、上游边界与分层验收

Colorful-U1 的背景必须保留“全局模式 → per-head 模式 → 七色固定拓扑 MVP → Source Graph”的演进。第一版实机结果只支撑一台 ACE 固定服务一个工具头的七色协同；新架构的任意映射能力由代码、dry-run 和局部部署记录支撑，不能继承旧版本的全部实机结论。SnapAce / multiACE 的原有协议和框架也必须与个人重构分开。

后续迭代型或 Fork 项目应按版本分配证据，不能用最终架构覆盖真实开发历史。

## 9. 写作红线与反模式

### 9.1 默认禁止的夸大表达

作品集任何项目不得出现以下未经充分外部证据支持的表达：

- “生产级”“工业级稳定”“商业可用”；
- “海量并发”“百万连接”“极致性能”；
- “精通”“完全自主”“完整自治”；
- “从零独立开发整套系统”，但实际存在框架、上游或团队基础；
- “已完成 ROS2 driver”“已实现视觉抓取”，但仍处于路线阶段；
- 医疗辅助诊断、临床产品或疗效结论，但没有相应资质与临床验证；
- 自研编辑器内核、大型 IDE，但实际基于成熟编辑组件集成。

即使没有命中上述词语，任何超出证据的同义表达同样不允许。

### 9.2 常见内容反模式

- 用技术名词密度代替项目难点；
- 把最终方案直接当作最初问题，抹掉真实演进；
- 背景、方案和结果三段重复同一组功能；
- 将代码数量、commit 数量或测试文件数量直接解释为质量；
- 只写成功路径，不写失败恢复或当前限制；
- 将上游、团队或第三方能力并入个人贡献；
- 把规划技术放入当前 Stack；
- 在页面显示“禁止写……”“待修”“后续再补”等内部备注；
- 将项目原站的普通响应式缺陷、UI 小问题或部署杂项提升为作品集核心边界；
- 为了填满所有详情组件而编造 `architecture`、`audit`、`research` 或指标。

## 10. 从资料到页面的实施流程

后续每个项目按以下顺序推进：

### 阶段 A：发现

- 确认目录、仓库、分支、文档、上游和素材；
- 建立项目时间线与初步模块图；
- 标记需要用户确认但可以先通过仓库调查的问题。

### 阶段 B：审计

- 核对作者、代码、构建、测试、部署或实机记录；
- 运行安全且与核心声明直接相关的验证；
- 建立事实台账，区分当前、历史、线上、mock 与规划。

### 阶段 C：凝练

- 先写“起因—矛盾—演进—结果”的短版本；
- 再补关键决策、个人贡献、协作 / 上游与边界；
- 最后选择指标和 evidence，避免先选漂亮数字再倒推叙事。

### 阶段 D：素材

- 清点并筛选真实素材；
- 需要时运行项目或浏览部署页面获取新截图；
- 为每张素材指定字段、caption 和证据职责；
- 优化格式、体积和隐私信息。

### 阶段 E：实现

- 更新 `src/data/projects.ts`，必要时再扩展 `src/types/index.ts` 和详情组件；
- 不在 JSX 复制项目长文案；
- 首页、项目列表和详情页始终复用同一项目对象。

### 阶段 F：验收

- 核对事实、数字、日期、commit、链接和素材；
- 检查首页摘要与详情是否存在口径冲突；
- 检查轮播、图片放大、筛选、返回与外链；
- 执行 lint、build 和浏览器控制台检查；
- 验证桌面、平板与 390px 移动视口；
- 最后进行一次夸大表达、上游归属和未来时态审查。

## 11. 项目完成定义

只有同时满足以下条件，项目内容才算完成：

### 事实

- [ ] 核心声明能够定位到证据；
- [ ] 当前与历史状态已区分；
- [ ] 代码、构建、测试、dry-run、部署和实机没有混用；
- [ ] 个人、团队、上游和第三方贡献边界清楚；
- [ ] 未来路线没有写成已交付成果；
- [ ] 最强指标包含条件和解释范围。

### 叙事

- [ ] 摘要能在两句话内说明项目矛盾、方案和结果；
- [ ] 背景是项目真实起点，不是行业套话；
- [ ] 方案说明关键决策，而非仅列功能；
- [ ] highlights 能定位到个人动作；
- [ ] 如确有 boundaries，它只阻止必要误读，不罗列缺失功能或内部待办；
- [ ] 详情长度由项目复杂度决定，没有为统一版式灌水。

### 素材

- [ ] cover 能代表项目；
- [ ] 首屏轮播只保留高信号素材；
- [ ] 正文图片与其支持的内容相邻；
- [ ] alt、caption、版本和环境信息准确；
- [ ] 无敏感信息，格式与体积适合 Web；
- [ ] 桌面与移动端裁切可接受，图片可点击放大。

### 工程

- [ ] 数据符合 `Project` 类型；
- [ ] 分类、状态、证据等级和链接类型正确；
- [ ] 首页、项目列表和详情页事实一致；
- [ ] 所有资源与外链正常；
- [ ] lint、build、控制台和关键交互通过；
- [ ] 没有新增写死项目数量或固定叙事的页面文案。
